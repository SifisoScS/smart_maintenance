## ğŸ§© **Design Patterns in Action**

| Pattern                       | Description                                                                  | Implementation Example                           |
| ----------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------ |
| **Factory**                   | Creates specialized maintenance request objects (Electrical, Plumbing, etc.) | `MaintenanceRequestFactory.create_request(type)` |
| **Repository**                | Abstracts data access for modularity and testability                         | `UserRepository`, `RequestRepository`            |
| **Observer**                  | Sends automatic notifications when requests change                           | Notifies Admin/Technician of status updates      |
| **Strategy**                  | Supports different notification types (Email, In-App)                        | `EmailNotification` vs `InAppNotification`       |
| **Singleton**                 | Manages centralized app configuration or database session                    | `ConfigManager` or `DatabaseConnection`          |
| **Dependency Injection (DI)** | Passes services into controllers dynamically                                 | Injecting `MaintenanceService` into Flask routes |

---

## ğŸ§© **Example Workflow**

1. **A user submits** a maintenance request via the Blazor interface.
2. **Flask backend** creates a corresponding `MaintenanceRequest` object using the **Factory pattern**.
3. The **RequestRepository** saves the request to the database.
4. Once assigned, the **Observer pattern** triggers a notification to the selected technician.
5. The technician updates progress, which is reflected instantly on the adminâ€™s dashboard.
6. The **Strategy pattern** determines whether to send an email or in-app update.
